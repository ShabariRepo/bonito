⌘
CLI
bonito-cli
+

The harness shells out to bonito commands to manage agents, knowledge bases, and credentials. Authenticated by a token in the environment.

What it does: deploys agents from YAML, uploads KBs, lists models, fetches logs.

Endpoint surface: talks to /api/* on the Bonito backend.

Token: bp- (personal access) or bj- (project-scoped).

$ bonito deploy -f bonito.yaml
$ bonito agents list
$ bonito kb upload my-docs ./docs/*.md
⇋
HTTP API
OpenAI-compatible
+

The harness makes direct HTTP calls to Bonito's gateway, using the OpenAI SDK or plain fetch. This is how production apps actually run AI workloads through Bonito.

What it does: chat completions, image generation, video generation, embeddings — routed across providers.

Endpoint surface: POST /v1/chat/completions, /v1/embeddings, /v1/images/generations, /v1/videos.

Token: bn- (gateway) or bp-.

POST https://api.getbonito.com/v1/chat/completions
Authorization: Bearer bn-...
{ "model": "gpt-4o", "messages": [...] }
⚙
MCP Server
bonito-mcp
+

For harnesses that speak the Model Context Protocol, Bonito ships an MCP server. The harness calls Bonito tools natively as if they were built-in functions.

What it does: list providers, list models, create agents, create KBs, fetch logs and costs — all as tool calls in the harness's own loop.

Configured in the harness's MCP settings with BONITO_API_KEY in env.

{
  "mcpServers": {
    "bonito": {
      "command": "uvx",
      "args": ["bonito-mcp"]
    }
  }
}
02 / SET UP

From zero to first agent.

Linear flow. Each step expands to show the actual command or call the harness would make.

1
Authenticate
Token in environment
+

The harness reads BONITO_API_KEY from .env. No browser login required — the token is the auth.

Token prefixes signal scope: bn- for gateway-only, bp- for everything-as-this-user, bj- for project-scoped.

# .env
BONITO_API_KEY=bp-...
BONITO_API_URL=https://api.getbonito.com
2
Connect a provider
OpenAI, Anthropic, Vertex, Bedrock, Azure, Groq
+

One-time setup per org. Each provider's credentials live in Bonito's secrets vault, never in your code.

Once connected, every Bonito agent and every gateway call can route to that provider's models. Adding a new provider later is a config change, not a code change.

$ bonito providers connect openai \
    --api-key sk-...
3
First gateway call
Verify the auth + routing works
+

The harness fires a minimal chat completion. If it comes back, the auth, the providers, and the routing are all working end to end.

curl -X POST https://api.getbonito.com/v1/chat/completions \
  -H "Authorization: Bearer $BONITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role":"user","content":"hello"}]
  }'
4
Deploy an agent
From YAML, idempotent
+

The harness writes a bonito.yaml, runs bonito deploy, and the agent is live. Re-running the deploy is idempotent — edit the YAML, ship again.

version: "1"
name: my-first-agent
agents:
  helper:
    display_name: "Helper"
    model_id: gpt-4o
    system_prompt: |
      You are a concise, factual assistant.
    tools:
      mode: allowlist
      allowed:
        - kb_search
        - http
5
Execute the agent
From the harness, from prod, from anywhere
+

The harness calls the agent execute endpoint with a message. The agent runs through its configured tools and model on the Bonito side. Every call is logged, costed, and observable.

POST /api/agents/{agent_id}/execute
{ "message": "what time is it in Tokyo?" }
03 / AGENTS

What runs when an agent executes.

A request enters the agent boundary. Inside that boundary: a system prompt, configured tools, a session, and a gateway hop to a provider. Expand any node to see what it does.

Request
Harness → /api/agents/{id}/execute
▼
A
Agent boundary
System prompt + config
+

Each agent carries its own system prompt, model selection, model config (temperature, max_tokens), tool policy, and optional knowledge-base bindings.

The agent boundary is what gives the harness a stable contract: one stable agent ID, one stable behavior, regardless of how providers, models, or routing change underneath.

S
Session memory
Per-conversation context
+

The agent can run with optional shared conversational memory across turns. Memory is tier-gated and per-session, so a chat-style harness can keep context across requests without rebuilding state.

For persistent context that survives sessions, agents can use long-term agent memory backed by vector search.

▼ tool calls
kb
search_kb
RAG over linked KBs
+

Searches one or more knowledge bases the agent is bound to. Returns cited passages the agent can use in its answer.

Vector similarity via pgvector HNSW. Embeddings handled at ingest time, transparent to the harness.

⇨
http
Outbound HTTP calls
+

Agent makes outbound HTTP requests to URLs explicitly listed in its http_allowlist. SSRF protection and default-deny are baked in.

Useful for calling your own backend, third-party APIs, or webhook-style integrations.

⇉
invoke_agent
Agent-to-agent delegation
+

An agent can call another agent. Useful when one agent is a generalist and another is a specialist — the generalist hands off without leaking that complexity to the caller.

Delegation calls are logged in the parent's execution trace, so the harness gets a clean breadcrumb of who called whom.

▼
G
Gateway
Multi-provider routing
+

The agent's model call lands at the gateway. The gateway picks the right provider (the org's configured credentials), enforces policy (model allow-lists, spend caps), and handles cross-provider failover when things go wrong upstream.

The agent doesn't know which provider answered. It just gets a response. Provider-switching is operational, not architectural.

P
Provider
OpenAI, Anthropic, Vertex, Bedrock, Azure, Groq
+

The actual model serves the inference. Cost is captured per-request, tokens are counted, latency is measured.

If the provider fails (rate limit, timeout, model unavailable), the gateway routes to an equivalent model on another provider. The harness sees a successful response either way.

04 / GATEWAY

What the gateway does for you.

A thin wrapper that handles the unglamorous operational problems so the harness doesn't have to. Tap any to expand.

R
Routing
Model → provider, per-request
+

The gateway maps a model name to the right provider based on your org's connected credentials and any routing policies attached to the key.

Routing strategies available: cost-optimized, latency-optimized, balanced, failover-first, A/B test. Configured per-key or per-policy.

F
Failover
Cross-provider retry
+

When a provider fails — rate limit, timeout, model not available — the gateway automatically retries on an equivalent model from another provider.

The harness sees a successful response. The failed attempt is logged in the audit trail so ops can see what happened later.

$
Cost capture
Per-request, per-model, per-key
+

Every call writes a row: provider, model, tokens in, tokens out, latency, cost. Searchable, exportable, attributable to a key, a project, or a team.

Real-time dashboards aggregate the rows so you see spend trends, model efficiency (cost per 1K tokens), and per-customer attribution as it happens.

L
Rate limits
Per-key, per-org, per-agent
+

Each gateway key has its own RPM ceiling. Each agent has its own RPM ceiling. Each org has a tier-based total.

Breaching a limit returns a clean 429, not a provider-specific error. The harness handles one error shape regardless of which provider was being routed to.

05 / GOVERNANCE + OBSERVABILITY

What you see, what you control.

The layer that turns an experimental AI app into something you can ship to enterprise without rebuilding it. Not optional add-ons — these are part of the platform.

Trace
Breadcrumbs

Every agent run produces a structured trace: which agent was called, which tools it invoked, which sub-agents it delegated to, which provider answered, what the model returned, how much it cost. Visible in the dashboard as a tree, searchable by session, attributable to a request. The same data feeds debugging, billing, and audit.

Audit
Per-token, per-action

Every API call, agent execution, KB upload, provider connection, and policy change is written to an immutable audit trail. who did what when, with token attribution and IP. Retention windows are tier-gated; export is one click. SOC-2 evidence package generated on demand.

Policy
Model allow-lists + spend caps

Each gateway key carries its own model allow-list, spend cap, and rate limit. Build a key for an internal team with different ceilings than a key for a customer-facing app. Policy lives at the edge — the gateway enforces it before the call ever leaves Bonito's boundary.

Compliance
SOC-2, HIPAA, GDPR scanning

Compliance checks run across the org's connected providers and policy state, flagging gaps before they become findings. PII handling, retention, data residency, access scope — surfaced as actionable warnings, not buried in audit logs.

Cost
Real-time intelligence

Cost per 1K tokens by model, by request, by project. Forecasts based on trajectory. Optimization recommendations (cheaper equivalent model, prompt-length suggestions, routing tweaks). The data you need to defend an AI line item in a board deck.

Identity
SSO, SAML, RBAC

Okta, Azure AD, Google Workspace, custom SAML. Role-based access (admin, member, viewer), SSO enforcement, JIT provisioning. Break-glass admin paths for incident response. Token prefixes (bn-, bp-, bj-) carry their own permission surface, baked into the auth layer.

Approval
Human-in-the-loop queue

For high-risk agent actions, the platform pauses execution and surfaces an approval request. Risk assessment, auto-approve conditions, timeout handling, audit trail of every decision. Useful when an agent has access to spend money, send email, or modify production data.

Health
System Observer

A scheduled internal observer agent reviews aggregated metrics per org and produces structured health and governance findings — degraded provider, drifting cost, model deprecation impact, anomalous traffic. The observability layer that watches the observability layer.

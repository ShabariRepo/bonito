# `bonito-cli init-review` — CLI Code Review Setup

> **Status:** Spec / Design Doc  
> **Priority:** Secondary (fallback for users who don't want the GitHub App)  
> **Target:** `npx bonito-cli init-review`

---

## Overview

A single CLI command that sets up automated AI code reviews on any GitHub repo. It generates a GitHub Actions workflow file and configures it with the user's Bonito API key — no GitHub App installation required.

This is the **alternative path** for users who:
- Prefer keeping CI config in their repo (GitOps)
- Use GitHub Enterprise Server (where marketplace apps aren't available)
- Want more control over when/how reviews run
- Don't want to grant a third-party app access to their org

---

## Usage

```bash
# Interactive setup (prompts for API key if not in env)
npx bonito-cli init-review

# Non-interactive (reads from env)
BONITO_API_KEY=bn-xxxx npx bonito-cli init-review

# With options
npx bonito-cli init-review \
  --severity warning \
  --skip "*.lock,dist/**" \
  --max-files 50
```

## What It Does

### 1. Auto-detect GitHub repo

```
✔ Detected GitHub repo: octocat/hello-world (origin)
```

- Reads `.git/config` for the remote URL
- Parses owner/repo from HTTPS or SSH remote
- Fails gracefully if not a git repo or no GitHub remote

### 2. Prompt for API key

```
? Bonito API key (starts with bn-): bn-xxxxxxxxxxxx
  Or set BONITO_API_KEY environment variable
```

- Checks `BONITO_API_KEY` env var first
- If not set, prompts interactively
- Validates the key format (`bn-` prefix, minimum length)
- Does **not** store the key in any file — instructs the user to add it as a GitHub secret

### 3. Generate workflow file

Creates `.github/workflows/bonito-review.yml`:

```yaml
# Auto-generated by bonito-cli init-review
# Docs: https://docs.getbonito.com/code-review
name: Bonito AI Code Review

on:
  pull_request:
    types: [opened, synchronize]

permissions:
  pull-requests: write
  contents: read

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: getbonito/bonito-review-action@v1
        with:
          bonito-api-key: ${{ secrets.BONITO_API_KEY }}
          # severity-threshold: warning
          # max-files: 50
          # skip-patterns: "*.lock,dist/**"
```

### 4. Print next steps

```
✔ Created .github/workflows/bonito-review.yml

Next steps:
  1. Add your API key as a GitHub secret:
     Repo → Settings → Secrets → Actions → New secret
     Name:  BONITO_API_KEY
     Value: bn-xxxxxxxxxxxx

  2. Commit and push:
     git add .github/workflows/bonito-review.yml
     git commit -m "Add Bonito AI code review"
     git push

  3. Open a PR — Bonito will review it automatically 🐟
```

## CLI Options

| Flag | Default | Description |
|------|---------|-------------|
| `--severity` | `all` | Minimum severity to report: `all`, `suggestion`, `warning`, `critical` |
| `--max-files` | `50` | Skip review if PR has more than N files |
| `--skip` | `*.lock,dist/**` | Comma-separated glob patterns to ignore |
| `--model` | `auto` | AI model override (default: Llama 3.3 70B via Groq) |
| `--dry-run` | `false` | Print the workflow file to stdout instead of writing it |
| `--force` | `false` | Overwrite existing workflow file without prompting |
| `--help` | — | Show help |

## GitHub Action: `getbonito/bonito-review-action@v1`

The workflow calls a GitHub Action (separate repo: `getbonito/bonito-review-action`) that:

1. Fetches the PR diff using `$GITHUB_TOKEN`
2. Sends the diff to the Bonito API (`POST /api/v1/review`)
3. Posts the review as a PR comment using the GitHub API

### Action Inputs

| Input | Required | Default | Description |
|-------|----------|---------|-------------|
| `bonito-api-key` | Yes | — | Bonito API key |
| `severity-threshold` | No | `all` | Minimum severity to show |
| `max-files` | No | `50` | Skip if PR exceeds this |
| `skip-patterns` | No | `*.lock,dist/**` | Files to exclude |
| `model` | No | `auto` | AI model override |
| `fail-on-critical` | No | `false` | Fail the check if critical issues found |

### Action Outputs

| Output | Description |
|--------|-------------|
| `review-status` | `completed`, `skipped`, `error` |
| `critical-count` | Number of critical issues found |
| `warning-count` | Number of warnings found |
| `suggestion-count` | Number of suggestions |

## API Endpoint (for the Action)

The GitHub Action calls a dedicated review endpoint on the Bonito backend:

```
POST /api/v1/review
Authorization: Bearer bn-xxxxxxxxxxxx
Content-Type: application/json

{
  "diff": "...",
  "pr_title": "Add user authentication",
  "pr_url": "https://github.com/owner/repo/pull/42",
  "files": [
    {"filename": "auth.py", "additions": 50, "deletions": 10}
  ],
  "options": {
    "severity_threshold": "warning",
    "model": "auto"
  }
}
```

Response:

```json
{
  "status": "completed",
  "review": "## 🐟 Bonito AI Code Review\n\n...",
  "findings": {
    "critical": 0,
    "warning": 2,
    "suggestion": 5
  }
}
```

> **Note:** This endpoint is separate from the GitHub App webhook. It's authenticated via API key (not GitHub App JWT) and is used by the GitHub Action and any direct API consumers.

## Comparison: GitHub App vs CLI

| Feature | GitHub App | CLI / Action |
|---------|-----------|-------------|
| Setup effort | 1 click | 3 commands |
| Config files | None | `.github/workflows/bonito-review.yml` |
| Auth | GitHub App JWT | API key in GitHub Secrets |
| Repo access | Granted via installation | `$GITHUB_TOKEN` (already available) |
| Works on GHE Server | ❌ | ✅ |
| Customizable | Limited (roadmap) | Full workflow control |
| Rate limit | 5/mo free, unlimited Pro | Based on API key tier |
| Billing | Per-installation | Per-API-key |

## Technical Notes

- The CLI is a thin wrapper — all heavy lifting happens in the GitHub Action and Bonito API
- The CLI does **not** require authentication to generate the workflow file; the API key is only needed at runtime
- The `npx` distribution means zero install — just run it
- The workflow file is deterministic — running `init-review` twice produces the same file (idempotent)
